# 深度调研引擎

调用：`/research <对象>` 或 `/research <对象> | <侧重点>`

`$ARGUMENTS` = 用户要调研的对象。可带一个 `|` 分隔的侧重点（可选）。
示例：
- `/research CPO 共封装光学`
- `/research serenity 选股方法 | 她的核心逻辑和公开可查的代表性持仓`
- `/research 稳定币法案 GENIUS Act | 对算力板块的影响`

执行下面六步，**不要跳步**，也不要在产出里堆术语——目标是"深入浅出讲清楚"。

---

## 易懂铁律（与严谨同级 · 写正文与 notes 前逐条过）

把"深入浅出"当**硬规则**执行，不是口号。下面每条都可自检：

1. **术语首次出现 = 中文全称 + 一句人话**。任何专业缩写 / 行话第一次登场，先给中文全称，紧跟一句大白话说清"它是干嘛的"，必要时打个生活化比方；同一篇里二次出现可直接用简称。
   - ✗ "DSP 可省掉" ／ "微环 MRM（约 1–2 pJ/bit）"
   - ✓ "省掉了 **DSP**（数字信号处理器，专门给失真信号纠错的小芯片）" ／ "用**微环调制器 MRM**（在光路上给光信号'开关'的微型器件，省电但怕热）"
2. **先人话、后术语**：每个机制 / 每节开头，先用一句**不含专业词**的话讲清"是什么、为什么重要"，再展开技术细节——读者读完第一句就该有大致画面。
3. **每个核心机制配一个生活化类比**（搬家 / 水管 / 排队 / 快递 / 翻译…），给外行一个抓手，再回到准确表述。
4. **缩写密度闸**：同一段里挤进 **≥3 个未解释缩写**就是超载信号——拆段、解释，或改成转述。
5. **能大白话就不用行话**；非用不可的，按第 1 条解释。术语是为了精确，不是为了显得专业。

> 适用于 report.html 的正文、关键发现、TL;DR、「先说人话」「名词小抄」两块，以及 notes.md。严谨（来源、隐私、中立、证据强度）一条不减——易懂是**额外加层**，绝不用浅薄换严谨。

## Step 0 — 对象分型 + 语言路由

先判断 `$ARGUMENTS` 属于哪一类，研究角度随类型变（见 Step 3）：

| 类型 | 识别特征 |
|---|---|
| **概念** | 技术名词、行业术语、机制（如 CPO、稳定币、液冷） |
| **人物** | 人名 / 账号（如某分析师、某 KOL） |
| **方法论** | 一套做法（如 CANSLIM、某人的选股法、某种估值法） |
| **事件** | 已发生/正发生的具体事（如某次法案、某次发布会、某次暴跌） |
| **板块** | 一个产业/赛道（如光模块、机器人、算力） |
| **股票** | **单一上市公司股票**（含代码如 300750 / NVDA、明确个股名如"宁德时代""英伟达"、或"怎么看 XX 这只票 / XX 股价走势 / XX 还能涨吗"等针对单只标的的请求） |

混合型（如"serenity 的选股方法"= 人物+方法论）按主成分处理，兼顾另一维。

**股票类 → 改走专用流程（硬路由）**：一旦判定为「股票」类，**不走本 skill 的 Step 3，改为读取并完整遵循 `.claude/skills/stock/SKILL.md` 的流程**（它会复用本 skill 的检索纪律与 Step 4/5/6 产出机制，只把分析主体换成 13 周走势情景 + 条件式触发的 A–M 框架）。仍先做本 skill 的 Step 1（校正前提）。**只处理单一标的**：请求含多只股票时，取主标的、显式标注其余超出本次范围（无人值守不反问；交互式可反问澄清）。

**语言路由**：科技 / 学术 / 一手技术对象，英文原始来源优先检索（论文、官方博客、SEC filing、原始公告），再回中文消化。输出一律中文。

## Step 1 — 先校正前提，再研究（硬规则）

研究前先审 `$ARGUMENTS` 里有没有**错误前提或被预设的结论**。若有：
1. 先指出错在哪、正确的是什么；
2. 重述成一个干净的、可被事实回答的问题；
3. 再进入检索。

**绝不顺着错误前提往下做。** 例：用户说"研究 X 的最新持股"，但 X 的持仓本就无公开可核渠道——先点破这个前提缺陷，把问题改写成"X 公开可查的代表性持仓与方法论"，再研究。

## Step 2 — 检索（带纪律）

**信息源优先级（严格遵守，高优先级冲突时压过低优先级）：**
监管机构 / 交易所 ＞ 公司披露 / 官方 filing 公告 ＞ 权威媒体 ＞ 券商/机构研究 ＞ 社区/论坛。
科技与学术对象：英文一手来源优先。

**检索方式：**
- 把对象拆成多个子问题，**每个子问题单独检索**，不要把所有诉求塞进一条 query。
- 概念/事件类：约 8–15 次检索；人物/方法论/板块类：更多，直到每个子问题都有落地来源。
- 原始来源优于聚合器/二手转载。多个来源冲突时，**并列呈现冲突并标注**，不替读者武断下结论。
- 所有时间敏感信息（"最新""昨天""近期"）一律换算并标注**北京时间**。

**人物类对象的已知局限（必须照做）：**
X/Twitter 自 2023 起关闭免费 API 且强反爬，**无法稳定、完整拉取某账号实时时间线**。因此人物类调研：
- 主攻：该人的观点体系、方法论、被广泛引用/转载/整理成文的代表性言论、媒体与社区对其的分析。
- 不承诺：逐条原始推文、完整时间线。
- 在 report 与 notes 的人物章节**显式插入局限声明**（见 Step 4 模板的 `{{LIMITATION_BLOCK}}`）：本节基于公开可检索内容，非完整时间线，时效以检索日为准。

## Step 3 — 深度剖析（按类型组织，深入浅出）

通用要求：用**类比 + 具体案例**讲清楚机制，先讲"是什么/为什么重要"，再展开细节。每个论断尽量挂来源。**严格执行开头的「易懂铁律」**——尤其"术语首次出现给中文全称+一句人话"与"每个核心机制配一个生活化类比"，外行也能跟下来。

- **概念**：定义 → 底层机制/原理（打比方）→ 为什么现在重要 → **多个真实案例**（公司/产品/落地场景，具体到名字和数字）→ 关键玩家与人物 → 争议与风险 → 与五大关注板块的关联（光模块/机器人/算力/AI应用/航天，仅在确有关联时）。
- **人物**：是谁/背景 → 核心论点与方法体系 → 代表性公开言论（转述为主）→ 可核实的 track record → 外界与同行评价（正反都要）→ `{{LIMITATION_BLOCK}}`。
- **方法论**：解决什么问题 → 核心逻辑与步骤 → **一个完整 worked example**（拿真实标的/数据走一遍）→ 适用边界与失效场景 → 谁在用、效果如何。
- **事件**：时间线（北京时间）→ 起因 → 各方立场 → 已知影响 → 后续观察点。
- **板块**：边界定义 → 产业链拆解（上中下游+代表公司）→ 驱动因素 → 主要风险 → 当前关键变量。

**人物 / 网红 / 论断型对象的额外两节（必写）：**
- **红旗清单 + 现状**：把"如果此人是错的 / 在拉高出货 / 业绩注水，会出现哪些迹象"逐条列出，并逐条核对**当前是否已经出现**（删帖或改口、二级市场大额增发、被监管或交易所点名、重仓标的流动性异常、自述与可查记录冲突等）。比单纯罗列质疑更有判断力。
- **激励与反身性**：分析此人的**利益结构**与其结论的**自我实现效应**——例如大量粉丝跟单本身会拉抬其重仓的小市值标的，即便其真诚，这种反身性也会反过来美化其战绩。点破"业绩好"里有多少来自方法、多少来自跟风盘。
- track record 必须**同时呈现已知的亏损 / 回撤**，而非只列盈利；若公开渠道几乎找不到失败案例，**明示这一缺失本身就是一个信号**（可能存在删帖 / 幸存者偏差）。

**方法论型对象的额外要求（必写）：**
- **执行门槛 / 可复现性诚实评估**：说清这套方法要复现需要什么——背景、技能、数据源、时间投入；若其核心能力（如读懂某类非财务/工程信号、特定领域纵深）本身极难获得，**直说"框架可懂、但普通人基本不可复现"**，不把难以复现的方法包装成"可学"。
- **与既有范式对照**：把该方法放进谱系里——它与哪些已有打法/前人方法相似（如卖铲子、费雪式产业链调研 scuttlebutt、套利或周期框架等），**哪些是真创新、哪些是旧范式换皮**，不当成自成一派来写。

**正文收尾固定一节「下次看什么（监控信号）」**：对快速演进的对象（科技概念、板块、事件、活跃方法论），正文最后用一个 `<h2>` 加一节，列 **3–6 个具体、可跟踪的催化剂或数据点**——写清"谁 / 什么事件 / 大概什么时点 / 该看哪个数字"（例：某代工某节点量产是否兑现、某产品实际出货与营收占比、某指标的下次披露、下届行业大会）。目的是让整份报告落到"下一步该盯什么"，而不是停在静态描述。纯静态、不会变的对象（如已盖棺的历史概念）可省略此节。

## Step 4 — 产出三件套到主题文件夹

为本次调研建独立文件夹（路径见 CLAUDE.md 的 `ARCHIVE_ROOT`）：

```
<ARCHIVE_ROOT>/<YYYY-MM-DD>_<topic-slug>/
├── report.html      # 阅读型成品，用本 skill 的模板填充
├── sources.md       # 全部来源清单（研究痕迹）
├── notes.md         # 给 Obsidian 的精简 markdown（Step 5 复制）
└── data/            # 截图、结构化数据、原始抓取（如有，否则留空）
```

- `<YYYY-MM-DD>` 用北京时间当天；`<topic-slug>` 用对象的短横线英文/拼音 slug。
- **report.html**：读取 `${CLAUDE_SKILL_DIR}/templates/report.html`，把所有 `{{TOKEN}}` 替换为真实内容后写出。结构 token 见模板顶部注释。正文是已渲染好的 HTML 片段（标题用 `<h2>/<h3>`、正文 `<p>`、要点 `<ul>`、案例用 `<div class="case">`、风险/局限用对应 callout）。**正文中每个关键数据、每条有争议的结论，就地用 `<a href>` 挂到对应来源**，让读者点得到出处，而非只在末尾堆来源清单。
  - `{{PLAIN}}`「**先说人话**」：2–3 句**纯文本**，把本对象讲给**完全不懂这行的人**，零术语 + 一个生活化比方。与 TL;DR（分析师口吻、给懂行者的结论）分工不同——这块面向外行。**所有类型都填**。
  - `{{GLOSSARY}}`「**名词小抄**」：把正文出现的专业术语集中成"术语 → 人话"对照表，填 `<section class="glossary"><h2>名词小抄</h2><dl><dt>术语</dt><dd>人话解释</dd>…</dl></section>`。**概念 / 板块 / 方法论**类必填（一般 5–12 条）；**人物 / 事件**类填空字符串（同 `{{LIMITATION_BLOCK}}` 的留空惯例）。
- **sources.md**：每条来源一行，格式
  `- [类型] 标题 — 链接 — 北京时间日期 — 一句话摘要`
  `[类型]` 按优先级标注：`监管`/`披露`/`媒体`/`研究`/`社区`。按优先级从高到低排序。

**来源链接完整性（写 sources.md 与正文前逐条自检，硬规则）：**
- 每条来源 URL 必须**深链到具体文章/公告/报告页**，不得用网站首页或栏目页（如 `example.com/`、`example.com/news/`）充数。
- 写每条前核对：**标题/摘要描述的主体，与该 URL 实际指向的页面一致**——防止"标题写 A、链接指向 B"的张冠李戴。对不上就换源或删除。
- 若某条信息**只能找到首页/裸域名级链接**，不得把它当作可靠来源去支撑硬数据；依赖它的具体结论必须就地标注 **`（未独立核实）`**。
- **不在 sources.md 留正文未引用的孤儿来源**：每条来源都应在正文有对应使用。

**软来源限流（数据完整性，硬规则）：**
- 一个**具体硬数字/硬结论**若**仅有** `社区` 级、检索摘要、或首页/裸域名来源支撑，**不得写进"关键发现"**；要么放进正文并就地标 `（未独立核实）`，要么进一步找一手/权威源把它坐实。
- 不让单一弱来源的数字获得与一手披露同等的呈现权重。

- 同一来源最多直引一次、每条引文 < 15 词、加引号与出处；其余一律转述。**绝不复制整段原文，绝不复制歌词。**
- **维护总索引**：每次跑完，向 `<ARCHIVE_ROOT>/INDEX.md` **追加一行**（文件不存在则创建，表头：`日期 | 对象 | 类型 | 板块 | 一句话结论 | 文件夹`），新行置顶按日期倒序。让积累的调研可一览、可检索。**「板块」列**填五大常关注板块（光模块·机器人·算力·AI应用·航天）中确有关联者（可多个，用 `/` 分隔），无强关联填 `—`，不硬凑（判定标准同 Step 5 frontmatter 的 `related`）。

## Step 5 — 在 Obsidian 落一份（带双链）

把 `notes.md` 写到 Obsidian vault 的 Research 子目录（路径见 CLAUDE.md 的 `OBSIDIAN_VAULT`）：
`<OBSIDIAN_VAULT>/Research/<topic-slug>.md`

文件头 frontmatter：
```yaml
---
date: <YYYY-MM-DD>          # 北京时间（天）
created: <ISO8601 北京时间>  # 精确到分秒的生成时刻，如 2026-06-04T22:53:24+08:00；取当前时间：`date "+%Y-%m-%dT%H:%M:%S%z"`。站点同一天内按它降序（新生成的排最上面），必填
type: <概念|人物|方法论|事件|板块>
tags: [research, <对象关键词…>]
related: ["[[光模块]]", "[[算力]]"]   # 仅挂确有关联的板块
source_count: <条数>
archive: "<ARCHIVE_ROOT 下本主题文件夹的相对路径>"
---
```
正文：报告的**精简版**（核心结论 + 关键案例 + 风险/局限），不是全文照搬。正文里用 `[[]]` 双链关联到五大板块笔记及其他相关概念笔记——链接到尚不存在的笔记也没关系，Obsidian 会自动建占位，调研因此织进现有知识网。

## Step 6 — 自动同步上线（每次必做）

调研三件套写完、INDEX 更新后，**把本次成果提交并推送到 `main`，触发 GitHub Pages 自动部署**，让它出现在公开站 https://qiuyuanqr.github.io/searchX/ 。站点构建逻辑：CI 扫描 `research/` 下所有 `<YYYY-MM-DD>_<slug>/` 文件夹（须含 `notes.md`），渲染成信息流卡片 + 报告页。`push: branches:[main]` 即自动 build+deploy（见 `.github/workflows/deploy.yml`），**无需手动 dispatch**。

执行顺序（**不要跳步，构建不过不许推**）：

1. **构建门禁**：先在仓库根跑
   ```
   bun test && bun run web/build/cli.js
   ```
   测试须全绿，且输出的 `Built N entries` 里能看到本次主题（可 `grep '<对象>' web/dist/index.html` 自检）。**不过则先修，绝不推送会让站点构建失败的内容。**（`web/dist/` 已 gitignore，是构建产物，不进提交。）
2. **隐私终检（硬红线）**：推送=公开发布。推送前再扫一遍 `report.html` / `sources.md` / `notes.md`，确认**无任何用户私人信息**（持仓规模、负债、财务、健康、家庭）。涉及买卖一律条件化表述。有疑即停、问用户。
3. **精准提交**：只 add 本次主题文件夹与索引，**绝不用 `git add -A`**（避免把无关改动、本地文件、Obsidian 库一并提交；Obsidian 副本在仓库外，本就不该进 git）：
   ```
   git add research/<YYYY-MM-DD>_<topic-slug> research/INDEX.md
   git commit -m "research: <对象> 上线"
   git push origin main
   ```
4. **回报**：推送成功后，告诉用户已触发部署、给出公开链接（首页 + 本篇 `r/<YYYY-MM-DD>_<topic-slug>/`），并说明 Pages 构建通常需 1–2 分钟生效。

> 若当前不在 `main`、或工作区有不相关的未提交改动，先向用户说明再决定如何处理，不擅自合并或丢弃他人改动。仓库是公开站点源，推送即对外可见——这是用户已授权的常态发布流程，但隐私终检每次都要做。

## 贯穿全程的硬规则

1. **隐私（绝对红线）**：report.html / sources.md / notes.md 都是可导出/可分享文档。**永远不在其中写入用户的私人信息**——持仓规模、负债与还款压力、财务状况、健康、家庭等一律不写。涉及"是否该买/持有"用条件化表述（"若已持有…""若尚未建仓…"）保持中立、非个人化。私人背景只在对话里、且用户主动提起时才谈，绝不落进文档。
2. **数据完整性**：比较数字时让数据直接说话；若数据指向与既有结论相反，立即修正解读，绝不为维护结论而扭曲。
3. **不预设深层意图**：只调研用户指定的对象，不擅自外延扩展。
4. **时间**：所有时点用北京时间。
5. **不确定就标注**：来源缺口、时效边界、人物时间线缺失，都明确写出，不假装完整。
6. **版权**：见 Step 4 引用规则。
7. **中立措辞 + 证据—结论匹配**：报告通篇用**中立的分析师口吻**，不替对象背书、不渲染。禁用"封神/神话/最值得学/碾压/现象级/独门绝技"这类带崇拜或营销色彩的词；确需提到时，**归属给来源**（"被支持者称为…""某媒体形容为…"）而非作为客观陈述。更重要的是**让结论的强度与其证据的强度成比例**：一条结论若只靠单一来源、短窗口、或特定市场环境（如普涨期）支撑，就要在该结论旁点破这一点，不得让薄证据撑起强断言。"选股命中率高"与"方法长期有效/账户翻倍属实"是不同强度的命题，绝不混为一谈。
8. **易懂是硬指标（与严谨同级）**：见开头「易懂铁律」。术语首次出现必给中文全称 + 一句人话，每个核心机制配一个生活化类比，并填好 report 的「先说人话」「名词小抄」两块。严谨与易懂**同时达标**才算合格——不因"内容专业"就放任晦涩，也不为通俗牺牲准确与来源。
