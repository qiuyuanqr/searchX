# /factcheck 接入 akshare 行情核准 — 设计稿

**日期**：2026-07-02（北京时间）
**状态**：已对齐待实现
**背景**：`/factcheck` 的 Step 2.5 已把"美股盘前 / 收盘错时"这类时间陷阱讲透，但核对价格时的数据来源仍只有 WebSearch。而"XX 暴涨 / 暴跌 20%""XX 跌停""指数破 XX 点"恰恰是实时核查里最高频、最时间敏感的一类声明——网页快照与财经媒体转述常滞后或掐错时点。`/stock` 已有成熟的 akshare 核准方案（SKILL §2.3，含降级规则），把它以精简形式引入 factcheck。

## 目标

行情 / 价格类声明的核对，优先用 akshare 拉一手行情数据坐实，不再只靠 WebSearch 快照；akshare 不可用时自动降级，绝不因此中断核查。

## 方案：只改 `.claude/skills/factcheck/SKILL.md`（纯 skill 文案，无服务代码）

新增 **Step 2.6 — 行情数据核准（涉及个股 / 指数价格类声明时 · akshare 优先 · 可降级）**，紧跟 Step 2.5：

1. **触发条件**：待核声明含具体行情断言——个股涨跌幅 / 收盘价 / 涨停跌停 / 成交量 / 换手率、指数点位、板块涨跌。不涉及行情的核查完全不走此步。
2. **接口表（精简自 stock §2.3，A 股覆盖好）**：
   | 数据 | akshare 接口 |
   |---|---|
   | 个股日线（收盘价 / 涨跌幅 / 振幅 / 换手率） | `stock_zh_a_hist` |
   | 个股实时快照 | `stock_zh_a_spot_em` |
   | 指数日线 | `index_zh_a_hist` |
   用 Bash 跑 `python3` 调用；凡用 akshare 的数据标注「**接口名 + 拉取日期**」。
3. **与 Step 2.5 衔接**：先按 2.5 定位声明的时点与时段（盘前 / 盘中 / 收盘、事件日 vs 发布日），再拉**对应时点**的数据比对——akshare 解决"数据准不准"，2.5 解决"比的是不是同一个时点"，两步都要过。
4. **降级规则（照抄 stock 的措辞逻辑）**：akshare 未安装 / 网络失败 / 该市场无覆盖（港股、美股覆盖弱，美股价格仍走 WebSearch + 权威行情源 ≥2 源交叉）→ 自动降级，并在该数据处标「未经 akshare 核准」、酌情下调把握度。**绝不因 akshare 失败而中断核查。**
5. **裁定影响**：经 akshare 核准的行情比对可支撑强裁定（✅/🔴）；仅靠二手快照的行情比对，把握度封顶「中」。

同时在「能力局限」节补一句：行情核准仅 A 股有一手数据通道，港美股价格核对依赖公开行情页。

## 配套（一次性环境确认）

- Mac mini（check-runner 无人值守跑 `/factcheck` 的机器）需装 akshare：`python3 -m pip install akshare`。在 `setup-macmini.sh` 里加这一步（幂等），并在 check-runner README 环境一节提一句。MacBook 交互式用同理（stock 已在用，大概率已装，跑前自检即可——skill 的降级规则本身就兜住未装的情况）。

## 不做什么

- 不给 factcheck 引入 stock 那张完整接口大表（资金流向 / 融资融券 / 龙虎榜等）——核查只需"这只票那天到底涨没涨、涨多少"，两三个行情接口够了；真需要深挖的属于 `/stock` 的活。
- 不写任何服务代码、不动 runner / Worker。

## 验证

- skill 文案改完后，手动各跑一条：①A 股行情谣言（能走 akshare）②美股盘前价声明（走降级 + 2.5 时区纪律）③无行情内容的普通核查（确认不触发 2.6），检查笔记里的标注与把握度符合设计。
