# 事实核查 skill（/factcheck）— 设计稿

> 2026-06-25。新增第三个 skill：把看到的政策 / 公众号文章 / 热点 / 截图 / 新闻 / 链接丢给它，它回答三件事——**这事是真是假 · 来龙去脉是什么 · 有多可信**，并给出可追溯的判断依据。
> 一句话：**一个对标 research 严谨度的私人事实核查工具，产出只存本机 Obsidian，不上线、不碰现有线上系统。**

## 背景 / 动机

现有两个能力——`/research`（把一个**对象**讲透）、`/stock`（对一只**股票**做情景判断）——都不解决"判定一条具体消息真假"这件事。用户的实际需求：日常看到的政策解读、公众号文章、热点话题、新闻、群里转的截图，想随手丢进去，让它帮着判断真假、讲清原委、评估可信度。

这是一类**新任务**（判定声明真假），产出形态是**裁定 + 证据链**，与 research 的剖析报告、stock 的情景判断都不同。塞进 research 会搅浑它的定位、并让自动委派逻辑复杂化，所以独立成第三个 skill。

**为什么不上线**：核查内容是个人关注的私人材料，且时效性强、争议性高。现有公开站跑在 GitHub Pages、无法做访问控制，内容一旦发上去就全网可见。所以本期**只做 skill、结果存本机 Obsidian**，完全不碰公开站 / Cloudflare / 半自动流水线。将来若想要"仅自己可访问的私人核查站"，再作为独立增量另设计（技术路径是 Cloudflare + 访问控制，与本期无冲突）。

## 已拍板的决策

1. **独立第三个 skill，命令名 `/factcheck`**（与 research/stock 风格统一，描述用中文）。新增 `.claude/skills/factcheck/`，**不改 research / stock 任何一行**。最坏情况删掉该目录即回到现状。
2. **方案最轻：只做 skill。** 不建站、不动 Cloudflare、不动 `web/`、不动 `services/`、不进公开发布流水线。
3. **"轻"是指基础设施轻，核查严谨度对标 research。** 溯源、交叉验证、诚实留白、独立复核一样不少——否则违背"要有可靠判断依据"的初心。
4. **产出只存本机 Obsidian 的 `Factcheck/` 子目录，绝不写进 `research/`、不上线。** `research/` 是公开站数据源、CI 会扫；核查内容进了那里就等于公开。Obsidian 库在仓库之外，天然不进 git、不进站点构建，是最干净的隔离。
5. **复用 research 的检索纪律 / 来源分级 / 校正前提 / 独立复核思路**，不重造。
6. **裁定分两个独立维度**：事实裁定（内容真假）+ 可信度评估（这条消息值不值得信）。二者分开打分。
7. **隐私**：私人站不受公开站"隐私红线"约束（只有自己看），但仍不把无关第三方的隐私信息乱写进笔记。

## 与 /research、/stock 的边界

| skill | 处理对象 | 产出 | 去向 |
|---|---|---|---|
| `/research` | 一个对象（概念/人物/方法论/事件/板块） | 剖析型 HTML 报告 + 三件套 | 公开站 |
| `/stock` | 一只上市公司股票 | 13 周情景判断 + 条件触发 | 公开站 |
| `/factcheck` | **一条待核实的具体消息 / 声明 / 截图 / 链接** | **真假裁定 + 原委 + 可信度** | **本机 Obsidian（私人）** |

触发场景：用户给的是一条**待验证的内容**，问"这是真的吗 / 可信吗 / 到底怎么回事"。本期**不做** research 的自动委派（保持独立、简单），需要时直接 `/factcheck`。

## skill 工作流程（Step 0–5）

```
Step 0  输入识别        判定输入是 文本 / 图片 / 链接 / 混合，提取出"待核查的内容"
   ↓
Step 1  拆声明 + 校正前提  拆成可独立验证的原子说法；分清事实 vs 观点；先点破错误前提
   ↓
Step 2  溯源 + 交叉验证    每条说法回一手来源，按来源层级排，重要结论 ≥2 独立源互证
   ↓
Step 3  裁定 + 讲原委      两维度裁定（事实裁定 + 可信度）；梳理时间线与来龙去脉
   ↓
Step 4  独立复核          对支撑最终裁定的 1–2 个承重事实，换角度再核一遍，防被带偏
   ↓
Step 5  产出             写一份 Obsidian 笔记（结论先行），存 Factcheck/ 子目录
```

## 输入处理：三类，各有核查切入点

- **文本 / 话题**：直接把里面的说法拆开核查。
- **图片（截图）**——用户在对话里贴图，Claude 原生识别。核查三层：
  1. 图里的**说法和数据**（当文本核查）；
  2. **出处线索**：公众号名、媒体水印、可见日期、账号；
  3. **图本身可不可信**：是否有 AI 生成 / PS 痕迹、是否"老图新配"（真图被安到错误的时间地点事件上）。
  - 局限：本工具**无法做以图搜图 / 像素级取证**，图像真伪主要靠"图中可识别线索 + 声称语境是否自相矛盾"来判断，存疑时明确写"无法确证图片来源"。
- **链接**——用 WebFetch 抓正文。核查除说法本身外，额外评估：
  4. **发布者背景与立场**（这个号/站什么来头，有无明显倾向）；
  5. **标题与正文是否一致**（标题党 / 断章取义）。

## 裁定体系（两个独立维度）

**为什么分开**：一个平时可信的媒体偶尔也会发错（内容假但源可信），一个不靠谱的号偶尔也蒙对（内容真但源不可信）。混在一起打分会误导。

**A. 事实裁定**（内容本身，六档）：

| 裁定 | 含义 |
|---|---|
| ✅ 属实 | 主要事实经多源核实成立 |
| 🟢 大体属实 | 主干成立，个别细节有出入 |
| 🟡 半真 | 选择性呈现 / 缺关键背景，孤立看会误导 |
| 🟠 误导 | 用真实碎片拼成失真叙事 / 断章取义 / 旧闻新炒 |
| 🔴 不实 | 主要事实与可查证据矛盾 |
| ⚫ 无法证实 | 现有公开证据不足以判定，**悬而不决（不猜）** |

**B. 可信度评估**（这条消息值不值得信，高/中/低 + 一句话理由）：综合三因子——
- **来源层级**：监管/官方 ＞ 权威媒体 ＞ 机构 ＞ 自媒体/匿名；
- **证据强度**：一手 / 多源交叉 ＞ 单一弱源 / 传闻；
- **一致性**：与已知事实是否冲突。

## 产出：一份 Obsidian 笔记（不搞三件套）

只在 Obsidian 看，主产物就**一份 markdown 笔记**，来源直接内嵌，不另出 HTML / sources.md。

**路径**：`<OBSIDIAN_VAULT>/Factcheck/<YYYY-MM-DD>_<话题slug>.md`（独立子目录，与 `Research/` 分开）。

**frontmatter**：
```yaml
---
date: <YYYY-MM-DD>              # 北京时间（天）
created: <ISO8601 北京时间>      # 精确到秒的生成时刻
type: 核查
verdict: <属实|大体属实|半真|误导|不实|无法证实>
credibility: <高|中|低>
input_type: <文本|图片|链接|混合>
tags: [factcheck, <话题关键词…>]
related: ["[[光模块]]"]          # 仅确有关联时挂双链（五大板块或相关概念笔记）
source_count: <条数>
---
```

**正文结构（结论先行）**：
```
1. 结论先行    一行给出 事实裁定 + 可信度，让你一眼看完
2. 原委        时间线 + 来龙去脉 + 各方说法 + 这事为什么容易被误解 / 传开
3. 逐条核查    每条原子说法 → 裁定 → 证据 → 来源链接（就地挂 a 链接）
4. 可信度依据   来源层级 / 证据强度 / 一致性 三项分别说明
5. 局限        哪些没查到、时效边界、图片是否无法确证来源
```

用 `[[]]` 双链关联到相关板块 / 概念笔记，让核查也融入 Obsidian 知识网（链到不存在的笔记也没关系，Obsidian 自动建占位）。

## 复用 research 的什么 / 不重造

从 `.claude/skills/research/SKILL.md` 直接搬的规则：
- **来源优先级**：监管/交易所 ＞ 公司披露/官方 ＞ 权威媒体 ＞ 机构研究 ＞ 社区；科技/学术英文一手源优先。
- **检索纪律**：拆子问题、每个单独检索、原始源优于二手、冲突并列标注、时间一律北京时间。
- **校正前提**（research Step 1）：输入预设了错误前提时先点破，不顺着错的往下查。
- **独立复核思路**（research Step 5.5 的精神）：对承重事实换角度再核，防自我幻觉。本期是私人参考、不对外发布，**做轻量版**——只复核支撑最终裁定的 1–2 个承重事实，不搞 push 前的多 agent 对抗核验全套。

## 严谨性硬规则（写笔记前逐条过）

1. **禁编造**：查不到就写"信息缺口 + 应去查的位置"，绝不臆造。
2. **强标注**：每个关键事实标来源类型 + 发布日期（北京时间）。
3. **区分事实 / 观点 / 推测**：观点不做真假判定，只评其依据。
4. **证据强度匹配结论强度**：单一弱源 / 短窗口撑不起强裁定，旁注点破。
5. **冲突并列**：来源打架时并列呈现 + 标注，不替读者武断下结论。
6. **诚实留白**：证据不足就给"无法证实"，不为凑结论而猜。
7. **中立措辞**：分析师口吻，不渲染、不站队。

## 能力局限（在 skill 内明确声明）

- **无以图搜图 / 像素级图像取证**：图片真伪靠图中线索 + 语境矛盾判断，非技术取证。
- **WebSearch / WebFetch 为检索基线**，无法访问付费墙后内容、无法稳定拉取 X/Twitter 完整时间线（同 research 局限）。
- **核查是生成时刻的快照**：内容定格在生成那一刻，事态后续变化需重新跑。

## 验收方式

skill 无代码单测（像 research/stock 一样靠端到端实跑）。验收 = 开新窗口跑 `/factcheck <一条真实、可查证的消息>`，检查：
- 产出落到 `<OBSIDIAN_VAULT>/Factcheck/`，frontmatter 与正文结构齐全；
- 事实裁定与可信度分开给出、都有依据；
- 每条裁定挂得到来源；查不到的诚实标"信息缺口 / 无法证实"；
- 至少跑一个**文本**、一个**链接**、一个**图片**输入，确认三类都通。
