# 核查任务状态 + 结论回显 — 设计稿

**日期**：2026-07-02（北京时间）
**状态**：已实现上线（2026-07-02；Worker 版本 25c799be，待真机 E2E）
**背景**：手机核查入口目前"提交即黑盒"——提交成功后，任务是排队、在跑、失败还是完成，页面上完全看不到；结果要等通知邮件 + Obsidian Sync 才能看到。本设计让 check.html 直接显示任务状态，并在完成后回显一行结论，不用打开 Obsidian 就知道真假。

## 目标

1. check.html 显示最近任务列表：排队中 / 已完成 / 已失败（停止重试）。
2. 任务完成后，页面直接显示**一行结论**（裁定 + 一句话），完整笔记仍在 Obsidian。
3. 隐私不退让：结论只在作者自己的私密通道里流转（凭 CHECK_KEY 查询、KV 7 天过期）；通知邮件照旧不含内容明文；公开站其他页面不受影响。

## 现状（改动落点）

- Worker `/check` 路由（`services/intake-worker/src/check.js`）：提交（CHECK_KEY）/ pending + image + done（CHECK_RUNNER_SECRET）。任务对象 `{ text, link, status, createdAt, images }`，`status` 只有 `pending` / `done`。
- check-runner（`services/check-runner/src/`）：跑完 `claude -p "/factcheck …"` 后 markDone；毒任务达上限后退休（也走 markDone）。
- check.html（`web/src/check.template.html` + `assets/check-page.js`）：只有提交表单，无任务列表。

## 方案

### 1. Worker：任务对象加终态信息 + 新查询端点

- **done 端点扩展**：`POST /check/<id>/done` 接受可选 JSON body `{ outcome: "done" | "failed", summary: "<一行结论，≤200 字>" }`。缺 body 时行为同现在（`status: "done"`，向后兼容）。`outcome: "failed"` 时任务标 `status: "failed"`（退休任务用）。summary 存进任务对象，随 7 天 TTL 过期。
- **新端点** `GET /check/recent`：凭 `x-check-key`（作者浏览器，同提交端点的 CORS 处理），返回全部 `check:*` 任务的轻量视图：`{ id, createdAt, status, summary?, textSnippet }`。`textSnippet` = text 前 40 字（无 text 用 link 域名 / "N 张图"），够认出是哪条即可；绝不返回图片字节。
- 保持两把密钥的隔离：recent 只认 CHECK_KEY，pending/done 只认 CHECK_RUNNER_SECRET。

### 2. check-runner：结论经信号文件带回（沿用 `.parked.json` 的信号文件模式）

claude 子进程与 runner 之间用**本机信号文件**传结论：

- runner 为每条任务生成结论文件路径 `<tmpdir>/searchx-check/<id>/verdict.txt`，并在拼 prompt 时（`buildFactcheckPrompt`）附一句：「核查完成后，把一行结论（裁定 + 把握度 + 一句话真相）写到 `<路径>`，仅此一行，不含其他内容」。
- claude 正常退出后，runner 读该文件（读不到 = skill 没写，降级为无 summary，照常 markDone——**结论回显是增强，不是硬依赖**），随 done 请求上传 `{ outcome: "done", summary }`，然后随现有 cleanup 一并删除临时目录。
- 退休路径上传 `{ outcome: "failed" }`（无 summary）。

### 3. factcheck skill：无人值守节补一条

`.claude/skills/factcheck/SKILL.md` 的「无人值守运行」节加一条：prompt 中若指定了结论文件路径，产出笔记后把一行结论写到该路径（格式：`裁定（把握度）：一句话真相`，如 `不实（高）：该截图系 2023 年旧闻拼接，原事件与 AI 无关`）。交互式运行不受影响。

### 4. check.html：最近任务列表

- 提交表单下方加「最近核查」区：页面打开时凭已存的 CHECK_KEY 拉 `/check/recent` 渲染；提供手动刷新；页面可见期间每 20 秒自动刷新一次（有排队中任务时才轮询，全部终态即停）。
- 每条显示：提交时间（北京时间）、textSnippet、状态章（排队中 = 灰 / 已完成 = 绿 / 已失败 = 红）、summary（完成时）。
- 纯前端渲染逻辑抽成可测纯函数（对齐 `check.js` / `check-page.js` 现有分层）。

## 不做什么

- 不显示"核查中"细分状态（runner 是批处理，任务从取走到 markDone 无中间状态可上报；引入"claiming"状态会带来锁与超时复杂度，先不做——排队中 + 60 秒轮询间隔下，体感够用）。
- 不做结果推送（邮件已有；页面轮询够用）。
- 不改 Obsidian 笔记（完整产出不动，页面只回显一行）。

## 测试与部署

- 单测：Worker（recent 端点鉴权 / done 带 body / 向后兼容无 body）、runner（读结论文件 / 文件缺失降级 / 退休带 outcome）、前端纯函数（列表渲染 / 状态章 / 轮询停止条件）。
- 部署顺序：先部署 Worker（向后兼容，旧 runner 照常工作）→ push 仓库（Pages 自动出新 check.html；Mac mini 自动拉新 runner）。Worker 需手动 `bun x wrangler deploy`。

## 风险

- summary 经 prompt 指令由 skill 写文件，模型可能偶尔漏写 → 已按"读不到就降级为无 summary"设计，不影响主流程。
- `/check/recent` 是新的对外读端点 → 只回轻量字段、凭密钥、有 CORS 限定 origin，与提交端点同等防护；KV list 成本与 pending 端点同量级。
