# 授权用户自助调研、自动放行 — 设计稿

> 2026-06-23。把"每次提交都要作者手动贴 `approved`"改成"白名单授权用户自助提交、系统自动跑"。
> 一句话：**给信任的人发一条专属提交链接，他用链接提交即自动跑研究，作者不再逐条审批。**

## 背景 / 动机

现状流程（M2a + M2b）：站内表单 → intake-worker（Turnstile + 校验 + 限频）建一个 `pending` GitHub Issue + KV 私存提交者邮箱 → **作者收到邮件、手动去 GitHub 给 Issue 贴 `approved`** → runner（Mac mini，每 5 分钟）扫到 `approved` 自动跑 `/research` → 上线 + 发信。

痛点：下游早就是全自动的，**唯一的人工卡点就是"去 GitHub 贴 approved"那一下**。站点本来是给自己人用的，作者不想每条都亲自放行。

本设计去掉这个卡点：用"授权白名单"替代逐条审批。

## 已拍板的决策

1. **验证方式：邮箱专属口令 / 链接**（不做登录页、不做会话、不给 Worker 配发信通道）。管理员在设置页加一个邮箱 → 系统生成该人专属 token → 管理员把带 token 的链接私发给本人。token 即凭证，邮箱由 token 映射决定（用户不在表单里输邮箱），无法冒充。
2. **公开提交：锁死。** 没有有效 token 不能提交，站点只对持链接的人开放。新人加入 = 管理员加邮箱 + 发链接。
3. **安全初筛红旗 → 降级人工。** 内容命中现有注入初筛（"忽略以上指令" / shell 命令 / `.env` 等）时**不自动放行**，降级为 `pending` 等作者手动批；其余全自动。理由：`/research` 是全权限 headless 跑用户内容，自动化后没人再人工核对红旗，保留这一道针对"可疑件"的保险。
4. **每授权邮箱每日上限**：默认 **5 篇/天**（可配），纯防链接泄露后被刷爆 Claude 额度，正常自己人够用。
5. **链接永不过期，仅手动撤销 / 轮换。** 朋友把链接留着即可永久免登录使用；唯一失效途径是作者在设置页撤销或轮换。
6. **管理员 = 唯一一把 `ADMIN_KEY` 的持有者（只有作者本人）。** 管理页是纯密钥闸：未输对密钥前什么都不显示、什么都干不了；真正的鉴权全在 Worker 服务端逐次校验。管理凭证与朋友的提交 token 完全隔离，提交 token 对管理接口无效。

## 总体流程

```
管理员在设置页填邮箱 someone@x.com
        ↓ Worker 生成专属 token，返回链接 https://<站点>/submit?k=<token>
管理员把链接私发给本人（微信/私聊，系统不发邮件）
        ↓
本人打开链接 → 页面调 /verify?k= 确认有效 → 回显"你已被授权（s***@x.com）"
        ↓ 填题目 / 侧重点 / 留言（无邮箱字段、无人机验证）提交，带上 k
Worker：invite:<token> 反查邮箱 → 校验内容 → 算安全红旗
        ├─ 无红旗 → 建【approved】Issue（正文标注"来自已授权用户，自动放行"）
        └─ 命中红旗 → 建【pending】Issue（正文标注"已授权但命中初筛，降级人工复核"）
        ↓ 仍把真实邮箱存 KV sub:<issue 号>（runner 取来发信，不变）
runner 每 5 分钟扫 approved → 自动跑 /research → 上线 → 发信  ← 完全不用改
```

**关键好处：runner 一行都不用改。** 它本来就自动处理 `approved`，本设计只是让 Worker 在源头直接贴 `approved`。

## 数据模型（Cloudflare KV `INTAKE_KV`，无新存储）

| 键 | 值 | 用途 |
|---|---|---|
| `invite:<token>` | `<email>` | 提交时用 token 反查邮箱（凭证查找）；**不设 TTL，永不过期** |
| `allow:<email>` | `{ token, addedAt }`（JSON） | 设置页列出授权人 / 撤销；**不设 TTL，永不过期** |
| `sub:<issue号>` | `<email>`（沿用，60 天 TTL） | runner 取邮箱发信，不变 |
| `rl:email:<email>:<日>` | 计数（沿用，调整上限） | 每邮箱每日额度 |

- 撤销一个人 = 删 `invite:<token>` + `allow:<email>`，口令立即失效。
- token：16 字节 `crypto.getRandomValues` → 32 位十六进制随机串。
- `allow:` 与 `invite:` 双向各存一份：列表按 `allow:` 前缀 list；提交按 `invite:` 直查，均 O(1)/一次 list。

## Worker 端点

改造与新增都在 `services/intake-worker/`。

### `POST /`（提交）— 改造
- 入参改为 `{ k, title, focus, message }`（**去掉 email、去掉 turnstile**）。
- `invite:<k>` 反查邮箱；查不到 → `403 unauthorized`。
- 用现有 `validateSubmission` 校验 title/focus/message（去掉 email 校验分支）。
- 现有 `screenSubmission` 算红旗。
- 每邮箱每日额度（`peek/commitRateLimit`，邮箱来自映射）；超限 → `429`。
- 建 Issue：`flags` 非空 → label `pending`；为空 → label `approved`。正文按两种情形分别标注来源。
- `sub:<number>` 存真实邮箱（不变）。

### `GET /verify?k=<token>` — 新增
- 有效 → `{ ok:true, email: maskEmail(email) }`（回显打码邮箱，供前端确认；不泄露完整邮箱）。
- 无效 → `{ ok:false }`。

### `GET|POST /admin/*` — 新增（管理员）
用一把 **`ADMIN_KEY`** 强密钥鉴权：请求带 `x-admin-key` 头，Worker 端**定长时间比较**校验；错误密钥按 IP 计数，超阈值临时 `429` 锁定（防暴力 / 防刷，详见安全设计）。
- `POST /admin/add` `{ email }` → 校验邮箱；若已存在则幂等返回原 token；否则生成 token、写 `invite:`+`allow:`。返回 `{ email, token }`（前端据此拼链接）。
- `GET /admin/list` → 按 `allow:` 前缀列出 `[{ email, token, addedAt }]`。
- `POST /admin/remove` `{ email }` → 读 `allow:<email>` 取 token，删两个键。
- （可选）`POST /admin/rotate` `{ email }` → 撤旧 token、发新 token（链接泄露时换一条，不必删人）。

### `GET /sub/<n>` — 不变

## 前端（`web/`，走 CI 自动部署）

### `submit` 页改造
- 从 URL 读 `?k=`；先 `GET /verify?k=` 确认 + 回显被授权邮箱。
- **隐藏邮箱输入框**（邮箱由口令决定）、**移除 Turnstile 控件**。
- 提交时带 `k`；无/无效 `k` → 显示"需要专属链接才能提交，请联系管理员"，表单禁用。

### 新增设置页（`admin` 静态页）
- 首次输入一次 `ADMIN_KEY`（存浏览器 localStorage），之后所有请求带 `x-admin-key`。
- 功能：① 填邮箱→加人，生成并显示**可一键复制的专属链接**（`<本站 base>/submit?k=<token>`，链接由前端按自身 origin/base 拼）；② 看授权人列表（邮箱 / 加入时间 / 复制链接 / 撤销）；③ 撤销某人。
- 页面静态、随 CI 部署；所有操作调 Worker `/admin/*`。

## 安全设计

- **口令是 URL 承载凭证**，可能因转发 / 浏览器历史泄露 → ① 高熵随机；② 设置页**随时可撤销 / 轮换**；③ **每邮箱每日上限**限制泄露后的爆炸半径（最多被刷 N 篇/天就触顶）。
- **邮箱不再由用户输入**（来自 token 映射）→ 杜绝冒充他人、杜绝往邮箱字段塞注入。
- **红旗降级**（决策 3）：可疑件不自动跑、留人工，守住"全权限 headless 跑用户内容"的唯一风险点。
- **隐私红线不变**：Issue 正文只放打码邮箱；真实邮箱只在 KV、用完即清（60 天 TTL）。

### 管理员页面访问控制（决策 6 展开）

"打开管理页"≠"拿到管理权限"。管理页是静态网页，**真正的锁在 Worker 服务端**：

- **管理员身份 = `ADMIN_KEY` 持有者**，只有作者本人。它与朋友的提交 token 是**两套完全独立的凭证**，提交 token 对 `/admin/*` 一律无效。
- **纯密钥闸**：管理页在输对 `ADMIN_KEY` 之前什么都不渲染、什么都不能做（只是一个密码框）。前端**不存在可被篡改的"我是管理员"开关**；每个管理动作都在 Worker 端逐次校验密钥。
- **`ADMIN_KEY` 强随机**（`openssl rand -hex 24`）、存 Worker secret、服务端**定长时间比较**（防计时侧信道）。
- **密钥走请求头 `x-admin-key`、不进 URL** → 不被 referer / 浏览器历史泄露（区别于提交 token 在 URL 里）。
- **失败尝试限流锁定**：对 `/admin/*` 的错误密钥尝试按 IP 计数（KV，复用限频套路），超阈值（默认 10 次/小时）临时 `429` 锁定——防暴力猜 + 防接口被刷。
- **管理页零第三方脚本**（防 XSS 偷走 localStorage 里的密钥）；站内任何位置**不放管理页入口链接**（不主动暴露——但这只是顺手，不当作真正防线，安全不靠藏网址）。

## runner 侧：不改

runner 仍只认 `approved` 未 `done`。Worker 直接贴 `approved` 后，runner 无感自动接手。查重 / park / 发信 / 锁全部沿用。**本设计对 `services/runner/` 零改动。**

## 新增配置（Worker）

| 名 | 类型 | 说明 |
|---|---|---|
| `ADMIN_KEY` | secret | 设置页鉴权密钥（`openssl rand -hex 24`） |
| `MAX_PER_EMAIL_PER_DAY` | var | 每邮箱每日上限，默认 5 |
| `ADMIN_MAX_FAILS_PER_HOUR` | var | 单 IP 错误密钥尝试上限，默认 10，超则临时锁定 |

`TURNSTILE_SECRET` 在授权路径不再使用（可保留或后续清理）。

## 部署注意（重要）

- **intake-worker 不随 CI 自动部署**：本功能大头在 Worker，改完要**作者手动重部署一次**（Cloudflare 控制台粘 `dist/worker.js` 最稳），并在控制台加 `ADMIN_KEY` secret。
- 前端（submit / admin 页）走 CI 自动上线。
- 上线顺序：先部署 Worker（带新路由 + ADMIN_KEY）→ 再让前端上线 → 设置页加第一个授权邮箱自测。

## 测试要点

- Worker（离线、注入 fetch/KV 假实现）：无 token→403；有效 token→approved；命中红旗→pending；超每日额度→429；admin 增/查/删；错 `ADMIN_KEY`→401，连续错超阈值→429 锁定；提交 token 调 `/admin/*`→401（凭证隔离）；`/verify` 有效/无效。
- 前端 `submit.js`：无 k→禁用并提示；verify 成功→回显 + 可提交。
- 构建 sanity：admin 页进产物、无报告残留等现有防护不破。

## 明确不做（YAGNI）

- 不做真正的登录态 / 会话 / Cookie / JWT（token-in-link 即凭证）。
- 不做 Worker 端发邮件（邀请链接由管理员手动转发）。
- 不做密码账号体系、不做 OAuth。
- 不做每人调研历史页 / 配额面板（将来再说）。
