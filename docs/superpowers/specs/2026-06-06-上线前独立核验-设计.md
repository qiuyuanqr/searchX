# 上线前独立对抗核验 — 设计稿

> 日期：2026-06-06（北京时间）  状态：已与作者对齐，进入实现

## 一句话

报告自动发布前，加一道**独立**（全新上下文、没参与写作）的对抗核验：派几个只读+联网的核验子 agent，**回到一手来源**把报告里承重的硬事实重核一遍；只有"拿得出对质证据的硬错"才挡上线，过不了走"划掉再发 / 搁置不发"兜底，搁置时邮件通知作者。

## 为什么做（不是重复劳动）

`research`（全程硬规则）和 `stock`（§3 事实核验 + M 节逻辑自查）内部已有大量核验，但**全是写作者在同一段上下文里自己查自己**。自查有结构性盲区：

- 它编出来的数字 / 日期，它自己是**相信**的，自查查不出来；
- 它引的来源到底支不支持那句话，写的时候没真回去重抓核对；
- 它对自己写下的结论有"承诺感"，倾向维护而非推翻。

换一个**没参与写作、任务就是来挑错**的全新上下文，恰好能逮住这一类——尤其"报告里那个硬数字 / 那句引述，回到一手来源对得上吗"。这是自查在结构上做不到的。

## 地基原则（死循环从根上不可能）

> **只有"客观、能拿出一手来源对质的硬错"才挡上线；一切主观的（判断、语气、不翻盘的逻辑跳跃）只标注或自动改，永远不挡。**

挡上线的集合因此是**有限、具体、可数**的（一篇里硬数字 / 硬日期 / 直接引述就那么十几二十条），不是开放式的"觉得不够好"——开放式才会无限循环。**走势判断、操作倾向是观点，不是可"核实对错"的事实，永远落软问题、永远卡不住上线。**

## 流程位置

插在 `research` 的 **Step 5（Obsidian 落地）之后、Step 6（push 上线）之前**，命名 **Step 5.5**。`research` 与 `stock` 都生效（stock 复用 research 的 Step 4/5/6，写进 research 即自动继承）。交互式跑和无人值守（runner）都走。

## 核验怎么做（独立性是关键）

派 **2–3 个核验子 agent**（只读 + 联网即可，不能改文件），**只给它们"成品报告正文 + sources 清单（含 URL）"，不给写作者的检索过程与推理**。每个子 agent 必须**回到一手来源重新抓取核对**，不许只凭报告内文判断（只重读不重抓 = 走形式）。三个视角：

| 核验员 | 盯什么 | 股票特化重点 |
|---|---|---|
| ① 数字与日期 | 每个关键硬数字 / 硬日期回到所标来源核对 | C 节财务表、估值、E 节事件日期 |
| ② 引述与来源指向 | 直接引述是否真在所标来源；链接是否指向所描述页面（防张冠李戴、防裸域名充数、防坏链） | 一手披露 / 公告链接 |
| ③ 内部一致 + 证据强度 | 自相矛盾（尤其 A 节结论 vs 后文）、结论强度是否超过证据、逻辑链断点 | J 逻辑链、A↔K 一致性 |

## 判定分档（只第一档挡上线）

**硬错（必须修，修不好不发）——判定门槛：必须拿得出"报告说 X / 来源说 Z / 来源 URL"对质证据**

- 承重硬数字 / 硬日期与所引一手来源对不上
- 引述张冠李戴，或来源里根本没有这句
- 撑硬结论的来源链接坏了 / 指向无关页面
- A 节方向判断依赖的那条事实根本不成立
- 报告内部自相矛盾且影响最终结论
- 隐私红线被破（写进用户私人信息）——本就是绝对红线

**软问题（标注 / 自动改，不挡上线）**

- 措辞强过证据 → 调弱 / 标"（单一来源·短窗口）"
- 撑硬结论的来源偏弱 → 降级为线索 / 加"（未独立核实）"
- 逻辑跳跃但不翻盘 → 记进 M 节
- 术语没解释、可读性 → 标出

## 控制流（封顶 2 轮，保证收敛）

1. 收到核验员的硬错清单后，主 agent **先逐条复核"这个错是真的吗"**——核验员可能读错来源、把对的当错的，这种**直接驳回，不"修"**（既防误伤把对的报告卡死，也帮收敛）。
2. 对**确认为真**的硬错：**定点修那几处**（改数字 / 换源 / 删站不住的引述 / 订正张冠李戴），**不重写全文**（重写又贵又可能引新错、不收敛）。软问题就地处理。
3. 修订后**只对改过的几处再核一遍**（改对了吗？引入新矛盾没？）。
4. 最多 **2 轮**。第 2 轮后仍有"确认为真且无法消解"的硬错 → 走兜底。

## 兜底（转满 2 轮仍有硬错）——分两种，保证上线的报告永远自洽

- **边角硬错**（不影响 A 节方向判断的事实）→ **划掉那条 / 就地标"（未核实）"，其余照常上线**。
- **撑着 A 节方向判断的硬错** → 不能删了支撑却留着结论（那报告自相矛盾）。要么**重写 A 节、下调置信度**让结论不依赖被删那条也站得住；若删了论点就垮 → **搁置不发（park）**。

## park（搁置不发）+ 邮件通知作者

`/research` 子进程在 runner 里**被剥掉了所有 `RUNNER_*` 机密**（runner.js 防提示注入：SMTP 密码 / PAT / 共享密钥都不进全权限会话）。所以**无人值守那条链路里 skill 自己发不了邮件**——park→邮件必须由**持有凭据的 runner 来发**，skill 只发信号。

**skill 端（park 时）：**

1. **不 push、不更新 INDEX**（绝不让有已知硬错的报告公开）。
2. 写信号文件 `research/.parked.json`：`{ topic, reason, unresolved: ["报告说X/来源说Z/URL", …], folder }`。
3. 交互式：直接把搁置原因 + 没解决的几条**摆给用户**（用户在场即通知）。
4. 无人值守：只写信号、不发邮件（拿不到凭据），由 runner 接手。

**runner 端（读到 `.parked.json`）：**

- 检查点放在 `runResearch` 之后、"无新文件夹"失败分支之前；先清信号（防泄漏到下一条 Issue）。
- **发邮件通知作者**（`composeParkNotice`，仅发作者、无 cc、不含私人信息）：主题 / 搁置原因 / 没解决的条目 / 本地草稿文件夹路径。
- 评论 Issue ⚠️ 点明"核验未通过已搁置待人工复核、未发布、未给提交者发信"。
- 贴 `done` **停重试**（park 会重复 park、白费额度；作者已收到邮件，人工接手）。
- `summary.parked++`；exit code 不因 park 置 1（park 非崩溃，邮件即信号）。

`research/.parked.json` 进 `.gitignore`，绝不入库；park 留下的本地草稿文件夹不 push、仅供作者本地查看。

## 成本与时间

范围**只框承重可核项**（A 节核心驱动 / 风险所依赖的事实 + 硬数字 / 硬日期 / 直接引述 / 来源指向，一篇十几二十条），几个子 agent **并行**，轮数**封顶 2**。墙上时间≈多跑一轮检索抓取，触发修订再加一点；无人值守那条异步、不在乎延迟。**范围框死 + 轮数封顶 + 并行**三条保证不会拖下去。
